# Exercise 02 — Security Software Discovery

## Objective

Execute a Security Software Discovery technique using Atomic Red Team
to simulate post-exploitation enumeration of installed security products.
Capture Sysmon and Windows Event Log telemetry, validate detection
coverage in Wazuh, identify gaps in existing detection rules,
create custom detection (if required), and document findings.

> Review the attack description and command.

**What this test does:**

This Atomic Red Team test performs **Security Software Discovery** by using
native Windows utilities to enumerate endpoint security controls such as
Microsoft Defender, Windows Firewall, Sysmon, and other installed security
products and services.

Key Commands Executed:

| Command | Purpose |
|---|---|
| `netsh advfirewall show allprofiles` | Enumerate Windows Firewall profiles and determine which profiles are enabled. |
| `netsh advfirewall firewall dump` | Dump complete Windows Firewall configuration and rule set. |
| `sc query windefend` | Check if Defender service (`WinDefend`) is installed and running. |
| `Get-Service \| Where-Object { $_.DisplayName -like '*sysm*' }` | Verify if Sysmon service is installed and running. |
| `Get-CimInstance Win32_Service -Filter "Description = 'System Monitor service'"` | Enumerate detailed Sysmon service information using WMI. |
| `tasklist.exe` | Enumerate all running processes. |
| `tasklist.exe \| findstr /i defender` | Search for Defender-related processes. |
| `tasklist.exe \| findstr /i virus` | Search for antivirus-related processes. |

**Attacker objective:**

Attackers commonly perform Security Software Discovery immediately after
gaining access to a system in order to:

- Identify installed antivirus and EDR solutions.
- Detect monitoring tools such as Sysmon.
- Enumerate Windows Firewall configuration.
- Understand endpoint defenses before executing malicious actions.
- Decide whether additional defense evasion techniques are required.

### Step 1 — Confirm Sysmon Captured the Activity

**Relevant Event IDs:**

| Event ID | Name | Generated By |
|---|---|---|
| EID 1 | Process Creation | `netsh.exe` (Windows Firewall enumeration commands) |
| EID 1 | Process Creation | `sc.exe` (Windows Defender service query) |
| EID 1 | Process Creation | `powershell.exe` (Get-Process, Get-Service, Get-CimInstance execution) |
| EID 1 | Process Creation | `tasklist.exe` (Running process enumeration and security process discovery) |
| EID 1 | Process Creation | `findstr.exe` (Filtering running processes for security products) |

> This Atomic test executes multiple Windows binaries to discover defensive software and system configuration.
> Each process generates an individual **Sysmon Event ID 1**.

**PowerShell — Collect Sysmon EID 1 Evidence:**

```powershell
# Set test execution time window
$testStartTime = Get-Date "2026-07-18 22:44:00"
$testEndTime = Get-Date "2026-07-18 22:46:00"

# Search for generated Sysmon events
Get-WinEvent -LogName "Microsoft-Windows-Sysmon/Operational" | Where-Object { $_.Id -eq 1 -and $_.TimeCreated -gt $testStartTime -and $_.TimeCreated -le $testEndTime -and $_.Message -match "netsh.exe|sc.exe|powershell.exe|tasklist.exe" } | Sort-Object TimeCreated | Select-Object TimeCreated, Id, Message | Format-List
```

> **Note**
>
> Sysmon logs individual process creations, not complete shell pipelines.
> Commands such as `tasklist.exe | findstr /i virus` generate separate events
> for `tasklist.exe` and `findstr.exe`. The complete command
> `tasklist.exe | findstr /i virus` will not be present in the **CommandLine**
> field of either process. The complete command is preserved in the
> **ParentCommandLine** field.
>
> This understanding is important when developing process creation detections
> because relying solely on the `CommandLine` field may overlook behaviours
> executed through shell pipelines.

### Detection Gaps

The following Windows Firewall discovery commands executed without a
dedicated detection:

- `netsh.exe advfirewall show allprofiles`
- `netsh.exe advfirewall firewall dump`
- `netsh.exe advfirewall show currentprofile`
- `netsh.exe advfirewall firewall show rule name=all`
- `netsh.exe firewall show state`
- `netsh.exe firewall show config`

### Custom Wazuh Detection

#### Step 1 — Check for Existing `netsh.exe` Rules

Before creating a custom detection rule, check whether Wazuh already
includes rules for `netsh.exe`.

On **ubuntu-server01**, run:

```bash
sudo grep -R -n -i "netsh" /var/ossec/ruleset/rules
```

The output shows that `0800-sysmon_id_1.xml` already contains rules
related to `netsh.exe`:

![Image](/images/atomic-red-team-exercises/02-security-software-discovery-t1518-001/26-create-detection.png)

View the relevant section of the rule file:

#### Step 3 — Create the Custom Detection Rule

> **Note:** Every Wazuh rule ID must be unique.
>
> Rule IDs **100000–120000** are reserved for custom local rules. In this lab,
> custom detections start at **100200** to keep all Purple Team rules within a
> dedicated range. Future detections can continue as **100201**, **100202**,
> and so on.

Add the following rule inside `local_rules.xml` and click **Save**:

```xml
<group name="sysmon,discovery,T1518.001,">
    <rule id="100200" level="10">
        <if_sid>92042</if_sid>
        <field name="win.eventdata.commandLine" type="pcre2">(?i)(advfirewall.*show|advfirewall.*dump|firewall.*show)</field>
        <description>T1518.001 - Windows Firewall Discovery via netsh</description>
        <mitre><id>T1518.001</id></mitre>
    </rule>
</group>
```

![Image](/images/atomic-red-team-exercises/02-security-software-discovery-t1518-001/33-create-detection.png)

> **Note:** Only matching elements such as `<if_group>`, `<if_sid>`, `<field>`, and
> `<match>` determine **when** a rule is triggered.
>
> Elements such as `<group>`, `<description>`, and `<mitre>` do **not**
> participate in rule matching. They only categorize the alert and add
> metadata after the rule has matched.

Now click on **Restart**:

![Image](/images/atomic-red-team-exercises/02-security-software-discovery-t1518-001/34-create-detection.png)

> Confirm the restart when prompted and wait for the Wazuh Manager to restart.

#### Step 4 — Verify Rule is Loaded

After the manager restarts, go to Wazuh Dashboard and navigate to:

`Left sidebar > Server management > Rules`

Search rule ID **100200** in search bar. If the rule appears in the results,
it has been loaded successfully.

![Image](/images/atomic-red-team-exercises/02-security-software-discovery-t1518-001/35-create-detection.png)

> **Important**
>
> When creating custom rules, use the decoded Wazuh field names rather than the
> field names shown in the Dashboard.
>
> For example, although the alert JSON displays fields such as:
>
> - `data.win.eventdata.image`
> - `data.win.eventdata.commandLine`
>
> the corresponding rule fields are:
>
> - `win.eventdata.image`
> - `win.eventdata.commandLine`
>
> The `data.` prefix is added by the indexer when storing events in
> OpenSearch and is **not** part of the field names used by the Wazuh rule
> engine.
