# Exercise 03 — Query Registry

## Objective

Execute a Registry Query technique using Atomic Red Team to simulate
post-exploitation enumeration of sensitive registry keys. Capture Sysmon
and Windows Event Log telemetry, validate detection coverage in Wazuh,
identify gaps in existing detection rules, create custom detection
(if required), and document findings.

### Step 1 — Review Test Details

![Image](/images/atomic-red-team-exercises/03-query-registry-t1012/02-attack.png)


![Image](/images/atomic-red-team-exercises/03-query-registry-t1012/03-attack.png)

> Review the attack description and command.

**What this test does:**

This Atomic Red Team test checks whether the **AlwaysInstallElevated** Windows
Installer policy is enabled for both the current user (`HKCU`) and the local
machine (`HKLM`). If enabled in both locations, standard users can install
MSI packages with **SYSTEM** privileges, creating a potential privilege
escalation path.

**Attacker objective:**

Attackers query the **AlwaysInstallElevated** registry policy to identify
misconfigurations that could allow a low-privileged user to execute a
malicious MSI installer with **SYSTEM** privileges, enabling local
privilege escalation.

|---|---|---|
| EID 1 | Process Creation | Each `reg.exe` invocation spawned by PowerShell |

## Part 4 — Detection Engineering

### Existing Wazuh Detection

No dedicated Wazuh rule exists for `reg.exe` querying sensitive
registry paths.

---

### Detection Gaps

The following Windows Firewall discovery commands executed without a dedicated detection:

- `reg query HKCU\SOFTWARE\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated`
- `reg query HKLM\SOFTWARE\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated`

---

### Custom Wazuh Detection

#### Step 1 — Check for Existing reg.exe Rules

Before creating a custom detection rule, check whether Wazuh already
includes rules for reg.exe.

On **ubuntu-server01**, run:

```bash
sudo grep -R -n -i "reg.exe" /var/ossec/ruleset/rules
```

The output shows that `0800-sysmon_id_1.xml` already contains rules related to `reg.exe`:

![Image](/images/atomic-red-team-exercises/03-query-registry-t1012/13-create-detection.png)

View the relevant section of the rule file:

```bash
sudo nl -ba /var/ossec/ruleset/rules/0800-sysmon_id_1.xml | sed -n '268,288p'
```

![Image](/images/atomic-red-team-exercises/03-query-registry-t1012/14-create-detection.png)

The built-in rule **92025** detects the execution of **reg.exe** and assigns it
**level 0**, which means it matches the event but doesn't generate an alert.
This rule can be used as the parent for our custom detection rule.

---

#### Step 2 — Navigate to Rules Editor

In Wazuh Dashboard, navigate to: `Left sidebar > Server management > Rules`

Click on **Manage rules files**:

![Image](/images/atomic-red-team-exercises/03-query-registry-t1012/15-create-detection.png)

Click on **Custom rules**:

![Image](/images/atomic-red-team-exercises/03-query-registry-t1012/16-create-detection.png)

Click the **Edit** (pencil) icon for `local_rules.xml`:

![Image](/images/atomic-red-team-exercises/03-query-registry-t1012/17-create-detection.png)

The editor opens with the contents of `local_rules.xml`:

---

#### Step 3 — Create the Custom Detection Rule

Add the following rule inside `local_rules.xml` and click **Save**:

```yaml
<group name="sysmon,discovery,T1012,">
    <rule id="100210" level="10">
        <if_sid>92025</if_sid>
        <field name="win.eventdata.commandLine" type="pcre2">(?i)AlwaysInstallElevated</field>
        <description>T1012 - Query Registry for AlwaysInstallElevated</description>
        <mitre><id>T1012</id></mitre>
    </rule>
</group>
```

![Image](/images/atomic-red-team-exercises/03-query-registry-t1012/18-create-detection.png)

Now click on **Restart**, confirm the restart when prompted and wait for
the Wazuh Manager to restart.

---

#### Step 4 — Verify Rule is Loaded

Search rule ID **100210** in search bar. If the rule appears in the results, it has been loaded successfully.
